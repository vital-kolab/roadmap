Backend architecture

Connect ROADMAP submissions to national-scale compute.

The public website can remain static and polished while the heavy work runs behind an API layer. ROADMAP should treat every model, dataset, and simulation as a versioned record that can be validated, queued, benchmarked, and summarized.

Layers

Build it in separable layers.

1. Public front end

GitHub Pages can host the landing page, audience portals, and submission UI. Forms call the backend API rather than trying to run compute directly.

  • Static HTML/CSS/JS
  • Accessible, role-specific pages
  • Submission previews and job-status cards
2. API gateway

A FastAPI service receives submissions, validates JSON schemas, handles authentication, and creates durable records.

  • /api/neural-hypotheses
  • /api/datasets
  • /api/jobs
  • /api/jobs/{id}/status
3. Registry database

PostgreSQL stores model metadata, dataset metadata, job manifests, permissions, benchmark selections, and audit logs.

4. Compute bridge

A controlled dispatcher converts validated simulation requests into SLURM jobs. The Alliance documentation identifies Slurm as the scheduler on its clusters, and jobs are submitted with account/resource options through commands such as sbatch.

5. Container runner

Each model run should use a pinned Apptainer/Singularity image or equivalent reproducible environment. That keeps user-submitted code away from the web server and makes runs auditable.

6. Benchmark + reporting

After a job finishes, a post-processing worker computes benchmark metrics and creates technical model cards, clinician-facing summaries, and community-readable explanations.

Minimal API contract

POST /api/jobs { "job_name": "Face-noise robustness sweep", "model_id": "RM-MODEL-0017", "task_family": "Faces", "perturbations": ["noise_sd_sweep", "feedback_ablation"], "benchmarks": ["human_behavior", "neural_reliability"], "resources": {"gpus": 1, "hours": 12, "memory_gb": 64} }

Minimal SLURM template

#!/bin/bash #SBATCH --job-name=roadmap_${JOB_ID} #SBATCH --account=${RAP_ACCOUNT} #SBATCH --time=12:00:00 #SBATCH --gres=gpu:1 #SBATCH --mem=64G module load apptainer apptainer exec roadmap_runner.sif \ python run_roadmap_job.py --job-id ${JOB_ID}
Security and governance

Guardrails ROADMAP should include from the beginning.

Separate trust zones

Keep public forms, backend validation, file storage, and compute execution in separate services with explicit permissions.

No arbitrary web execution

User code should never run on the web server. Only validated containers or reviewed pipelines should be dispatched.

Dataset permissions

Benchmark jobs should only access datasets that the submission is authorized to use.

Audit every result

Store model version, data version, container hash, job manifest, and benchmark code version for every output.

Role-specific outputs

Generate technical outputs for developers, evidence cards for clinicians, and accessible summaries for community users.

Human review gates

Require review before making claims public, especially for clinical or product-facing interpretation.

Recommended build order

Start with the submission UI and JSON schemas, then build a FastAPI prototype, then add a manual SLURM dispatcher, then automate status tracking and report generation. This avoids overbuilding before the scientific submission schema is stable.
