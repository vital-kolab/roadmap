Simulation workflow

Turn a research request into a reproducible compute job.

ROADMAP can present simulation as a simple public-facing request while running the actual workload on institutional or national compute. That means users interact with a guided form and results dashboard, while the platform handles packaging, queuing, tracking, and reporting.

Recommended flow

A seamless job pipeline for the public platform.

Step 1

Collect a structured request

Front-end users choose the task family, uploaded model or base model, perturbations, target datasets, and desired outputs.

Form output → job_spec.json
Step 2

Validate and package

A lightweight backend validates the request, assigns a job ID, stores metadata, and builds a reproducible manifest for execution.

  • schema validation
  • versioned config
  • user permissions
  • data access checks
Step 3

Dispatch to Compute Canada / Alliance

The backend submits a SLURM job using a standard runner, ideally inside Apptainer/Singularity containers so that environments stay reproducible.

sbatch roadmap_run.sbatch --export=JOB_ID=RM-2043
Step 4

Track queue and runtime status

The platform polls job state or listens for status updates and shows friendly states such as queued, running, benchmarking, complete, or needs attention.

Step 5

Benchmark outputs automatically

Finished outputs are stored, compared against selected benchmarks, and summarized into model cards, plots, and role-specific evidence summaries.

Step 6

Return results to the right audiences

The same result can produce a technical report for developers, an evidence summary for clinicians, and a plain-language explanation for community members.

Suggested architecture

What to build behind the GitHub Pages front-end.

GitHub Pages is a good public front-end, but the simulation service itself should live behind an API. A simple architecture is enough to get started.

1

Static front-end

Keep the public website on GitHub Pages. Use forms, dashboards, and submission flows as the main interface layer.

2

API service

Add a small backend, for example FastAPI, to receive submissions, authenticate users, and create job records.

3

Job database

Store requests, job state, model metadata, and result pointers in PostgreSQL or another durable database.

4

SLURM dispatcher

Use a controlled service account to turn validated requests into SLURM submissions on Compute Canada / Alliance.

5

Containerized runners

Package model execution and analysis scripts in Apptainer images so the same request always runs in the same environment.

6

Result storage + reporting

Write outputs to object storage or a shared results directory, then generate dashboard summaries and downloadable artifacts.

Why this feels seamless to users

Users should not see cluster details. They should only see:

  • a clean submission form,
  • clear job status updates,
  • estimated runtime,
  • email or dashboard notifications, and
  • results presented in the language appropriate to their role.

Best practices for ROADMAP

  • Use model templates so users are not starting from raw code every time.
  • Require metadata at submission time so benchmarking can be automated.
  • Separate public, restricted, and internal datasets clearly.
  • Keep a model registry with versions, benchmarks passed, and limitations.
  • Generate both technical reports and plain-language summaries from the same output.

Good minimum viable setup

Front-end: GitHub Pages.
Backend: FastAPI.
Database: PostgreSQL.
Compute: SLURM on Compute Canada / Alliance.
Containers: Apptainer.
Notifications: email + dashboard status updates.
